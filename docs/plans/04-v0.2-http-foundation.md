# Plan 04 (v0.2.0): HTTP foundation + source-read

**Status**: ready to implement
**Releases**: v0.2.0
**Goal**: Break the workspace-`IFile` wall identified in Plan 02. Add the
highest-value missing read tool (source code), plus a generic HTTP escape
hatch that future tools build on.

## Why this approach (vs. workspace-IFile sync)

Plan 02 documented a dead-end: most ADT services need an `IFile` synced from
the backend. We proposed building a sync helper (~60 lines) and layering
tools on top.

During v0.2 research we found a cleaner path: **call ADT REST endpoints
directly through Eclipse's public `IRestResource` / `ISystemSession`
infrastructure.** This bypasses the workspace entirely:

- No `IFile` creation, no workspace project pollution
- No "sync, wait, read" race condition
- Single API call → bytes
- Reuses Eclipse's destination, cookies, CSRF — same code path SAP's own UI uses

This is also strictly more useful long-term: it unblocks any future tool that
maps to a REST endpoint (where-used, transports, ATC, free SQL, etc., all in
v0.3+).

## Public API path verified (via bytecode inspection)

| Component | Bundle | Method |
|---|---|---|
| `AdtSystemSessionFactory.createSystemSessionFactory()` | `com.sap.adt.communication` | static, returns `ISystemSessionFactory` |
| `ISystemSessionFactory.createStatelessSession(String destinationId)` | same | returns `IStatelessSystemSession` (extends `ISystemSession`) |
| `AdtRequestFactory.createRequestFactory()` | same | static, returns `IRequestFactory` |
| `IRequestFactory.createInstance(Method, URI, IHeaders, IMessageBody)` | same | returns `IRequest` |
| `HeadersFactory.newHeaders()` + `.newField(name, values...)` | same | builds `IHeaders` |
| `ISystemSession.sendRequest(IProgressMonitor, IRequest)` | same | returns `IResponse` |
| `IResponse.getStatus()` + `.getBody().getContent()` | same | int + `InputStream` |

All in non-`internal` packages — no reflection. Confirmed implementations:
- `com.sap.adt.communication.session.AdtSystemSessionFactory` (public final, static `createSystemSessionFactory`)
- `com.sap.adt.communication.message.AdtRequestFactory` (public final, static `createRequestFactory`)
- `com.sap.adt.communication.message.HeadersFactory` (public final, static `newHeaders`/`newField`)

## Tools to ship in v0.2.0

### T8. `arc1_sap_http_get` — generic GET escape hatch

**Why first**: it's the smallest possible test of the whole API path. If this
works, every other HTTP-based tool works.

**MCP input**:
```json
{
  "destination": "A4H_001_marian_en_1",
  "uri": "/sap/bc/adt/discovery",
  "accept": "application/atomsvc+xml"
}
```

**MCP output**:
```json
{
  "status": 200,
  "contentType": "application/atomsvc+xml;charset=utf-8",
  "headers": { "x-csrf-token": "...", "etag": "..." },
  "body": "<?xml version=...>",
  "bodyTruncated": false
}
```

Body capped at 256 KB to avoid runaway responses. `bodyTruncated: true` if
hit. Power users can always paginate / use Accept-Range headers.

### T9. `arc1_sap_read_source` — typed wrapper for source GETs

**Why second**: it's the most-requested capability. Sits on top of T8 with
a friendlier schema and an Accept default.

**MCP input** — pass either the source URI directly OR the object URI +
type and we figure out the source-segment suffix:

```json
{
  "destination": "A4H_001_marian_en_1",
  "objectUri": "/sap/bc/adt/oo/classes/CL_ABAP_TYPEDESCR",
  "include": "main"
}
```

Or:
```json
{
  "destination": "A4H_001_marian_en_1",
  "sourceUri": "/sap/bc/adt/oo/classes/CL_ABAP_TYPEDESCR/source/main"
}
```

The `objectUri` form auto-appends `/source/main` (the common default). For
CLAS includes (`definitions`/`implementations`/`testclasses`/`macros`), pass
`include`. For other types where the source segment differs, use `sourceUri`
directly.

**MCP output**:
```json
{
  "status": 200,
  "uri": "/sap/bc/adt/oo/classes/CL_ABAP_TYPEDESCR/source/main",
  "source": "CLASS cl_abap_typedescr DEFINITION...",
  "byteCount": 12345,
  "truncated": false
}
```

Same 256 KB cap. Strings longer than that get a `truncated: true` flag.

### `AdtHttp` helper — central HTTP shape

One file consolidates the SAP API choreography so each tool stays tiny:

```java
final class AdtHttp {

    static final int MAX_BODY_BYTES = 256 * 1024;

    static Response get(String destinationId, String pathAndQuery, String accept) {
        IStatelessSystemSession session = AdtSystemSessionFactory
            .createSystemSessionFactory().createStatelessSession(destinationId);
        IHeaders headers = HeadersFactory.newHeaders();
        if (accept != null) {
            headers.setField(HeadersFactory.newField("Accept", accept));
        }
        IRequest req = AdtRequestFactory.createRequestFactory()
            .createInstance(IRequest.Method.GET, URI.create(pathAndQuery), headers, null);
        IResponse resp = session.sendRequest(new NullProgressMonitor(), req);
        return new Response(resp);
    }

    static final class Response {
        final int status;
        final Map<String, String> headers;     // flattened, last-wins
        final byte[] body;                      // first MAX_BODY_BYTES
        final boolean truncated;
        final String contentType;
        // build from IResponse
    }
}
```

## Tests (run after build + Eclipse restart)

Add to `scripts/smoke-test.sh`:

```bash
# T8: generic GET — call /sap/bc/adt/discovery
call_tool 10 arc1_sap_http_get \
  "{\"destination\":\"$DEST\",\"uri\":\"/sap/bc/adt/discovery\",\"accept\":\"application/atomsvc+xml\"}"
# Expect: status=200, body starts with "<?xml"

# T9: read source — well-known SAP class
call_tool 11 arc1_sap_read_source \
  "{\"destination\":\"$DEST\",\"objectUri\":\"/sap/bc/adt/oo/classes/CL_ABAP_TYPEDESCR\"}"
# Expect: status=200, source contains "CLASS cl_abap_typedescr"
```

## Risks + mitigations

- **CSRF token for GETs**: ADT requires a CSRF token for POST/PUT/DELETE,
  not GET. Pure GET path in v0.2 is unaffected. v0.3 (where-used uses POST)
  needs CSRF handling — Eclipse's `ISystemSession` handles this automatically
  on the same session, so this is not a real risk.
- **Large responses**: 256 KB cap with `truncated` flag prevents accidental
  flooding. A user requesting a 5 MB DDLS gets the first 256 KB + a clear
  signal to use pagination.
- **Wrong destination**: `sendRequest` will throw if destination is not
  registered / not logged in. Caught and returned as MCP `isError: true`.
- **Content type negotiation**: ADT source endpoints typically accept
  `text/plain`. If a tool sends the wrong `Accept` header, ADT returns 406.
  Surfaced cleanly to caller (status + body).

## Self-review

- **Why ship the generic `arc1_sap_http_get` at all?** Two reasons:
  (1) it's the smallest test of the API path — if it works, T9 works;
  (2) it's an escape hatch for power users to call any ADT endpoint without
  waiting for us to ship a typed wrapper. Future tools then become "typed
  wrapper for what `http_get` already does".

- **Why not include POST in v0.2?** POST adds body serialization, content-
  type negotiation, and CSRF handling — each is a small thing but together
  3x the surface. Ship GET first, prove it works in production, then add POST
  in v0.3 with where-used as the canonical use case.

- **Why not parse the response as XML/JSON?** Returning raw bytes-as-string
  lets the LLM (or the typed wrapper above us) parse however it wants. Adding
  XML/JSON parsing to the generic tool would lock us into a format and bloat
  the tool.

- **Is `arc1_sap_read_source` overlapping with the future `arc1_sap_object_info`
  workflow?** No — `arc1_sap_object_info` returns metadata (name/type/package),
  `arc1_sap_read_source` returns source text. Complementary, often called
  together by an LLM.

- **What if the source returned is in non-UTF-8 encoding?** ADT typically
  returns UTF-8 with explicit `charset=utf-8` in `Content-Type`. We honor
  the charset from the header; fall back to UTF-8 if missing. Test against
  a non-ASCII source (e.g. ABAP class with German comments) during smoke
  test.

**Verdict**: scope is right-sized. Two tools, one shared helper, no new
bundle deps, ~240 lines total. Implement.
