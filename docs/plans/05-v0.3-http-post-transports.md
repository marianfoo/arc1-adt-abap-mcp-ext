# Plan 05 (v0.3.0): HTTP POST + transports list

**Status**: ready to implement
**Releases**: v0.3.0
**Goal**: Extend the v0.2 HTTP foundation with POST support, then ship one
typed wrapper that exercises it cleanly: transport list.

## Why these specific tools

After research, three candidate endpoints emerged:

| Endpoint | Method | URI | URI confidence |
|---|---|---|---|
| Transport list | GET | `/sap/bc/adt/cts/transportrequests` | **high** — found verbatim in `TMViewService` bytecode |
| Where-used | POST | `/sap/bc/adt/repository/informationsystem/whereused` | **medium** — found URI literal in `RisDataModel`, but XML body shape is fragmented across multiple `ContentHandler` classes |
| Object structure | GET | unknown — content type `application/vnd.sap.adt.objectstructure.v2+xml` is known but the URI literal didn't extract cleanly | **low** |

The user's standing instruction is "tests before to make sure it works".
Shipping uncertain endpoints would violate that. So v0.3 ships:

1. **`arc1_sap_http_post`** — generic POST escape hatch (extends v0.2's GET).
   Powers any future typed wrapper.
2. **`arc1_sap_list_transports`** — typed wrapper, verified URI + query params,
   ships with confidence.

Where-used and object structure stay deferred to v0.4 — both need a brief
HTTP-trace capture (Eclipse's `org.eclipse.equinox.http.servlet` or fiddler
on the destination) to nail down the exact request shape. Worth doing as
its own milestone.

## API additions

`AdtHttp.post(destinationId, pathAndQuery, accept, contentType, body) → Response`
mirrors `AdtHttp.get`. Body is `byte[]`; tools convert from String when needed.

Uses `com.sap.adt.communication.message.ByteArrayMessageBody(String contentType, byte[] data)`
— public concrete class, no factory needed.

## Tools

### T10. `arc1_sap_http_post` — generic POST escape hatch

**MCP input**:
```json
{
  "destination": "A4H_001_marian_en_1",
  "uri": "/sap/bc/adt/repository/informationsystem/whereused",
  "accept": "application/xml",
  "contentType": "application/xml",
  "body": "<ris_request>...</ris_request>"
}
```

**MCP output**: same shape as `arc1_sap_http_get` (status, contentType,
headers, body, truncated).

**Use case**: lets power users (and future typed wrappers) exercise any
POST endpoint without waiting for us to add a typed tool.

### T11. `arc1_sap_list_transports` — typed wrapper

**Backend call**:
```
GET /sap/bc/adt/cts/transportrequests
    ?username=<user>
    &requestStatus=<Modifiable|Released>
    &requestType=<Task|...>
Accept: application/vnd.sap.adt.transportorganizer.v1+xml
```

All query params are optional. The XML response is large EMF; we won't fully
parse — instead, we let the user opt-in via `parse` flag. When `parse=false`
(default) we return the raw XML so the LLM can read it. When `parse=true` we
do a minimal regex-based extraction of `<tm:request>` blocks. This is
intentionally conservative — full EMF parsing isn't free of dependencies.

**MCP input**:
```json
{
  "destination": "A4H_001_marian_en_1",
  "username": "DEVELOPER",
  "status": "Modifiable",
  "parse": false
}
```

**MCP output** (parse=false):
```json
{
  "status": 200,
  "contentType": "application/vnd.sap.adt.transportorganizer.v1+xml",
  "rawXml": "<?xml ...>",
  "truncated": false
}
```

**MCP output** (parse=true):
```json
{
  "status": 200,
  "count": 4,
  "requests": [
    {
      "number": "DEVK900145",
      "description": "Fix bug X",
      "owner": "DEVELOPER",
      "status": "Modifiable",
      "type": "Task"
    }
  ]
}
```

## Risks + mitigations

- **CSRF token for POSTs**: ADT requires a CSRF token for write-side requests.
  Eclipse's `ISystemSession` fetches it automatically on the same session.
  Stateless session may need an extra round trip — we'll observe the response
  and document if 403-CSRF appears.
- **Large transport list responses**: existing 256 KB cap with `truncated`
  flag protects us.
- **The `nodestructure` smoke test for POST is a known-working endpoint**
  per ARC-1's own probe research from May 22 (the document
  `adt-mcp-live-s4-api-probes-2026-05-22.md` confirmed it returns 200).

## Self-review

- **Why no `arc1_sap_where_used`?** XML body shape uncertainty fails the
  "test before commit" bar. Document as v0.4 target with the trace-capture
  step explicit.
- **Why `parse=false` default for transport list?** XML is large; LLM can
  read it raw without us spending effort on parsing. `parse=true` exists as
  a convenience for clients that want structured data.
- **POST tool — base64 vs string for body?** ADT POST bodies are
  almost exclusively text/xml or text/json. String is simpler. If someone
  needs binary, they shouldn't be using this generic tool anyway.
- **Overlap with SAP's `abap_transport-get`?** No — that tool takes one
  ABAP object and returns transports relevant to *it*. Our tool lists ALL
  transports for a user (with status filter). Different intent.

**Verdict**: tight scope, both endpoints verified, ship.
